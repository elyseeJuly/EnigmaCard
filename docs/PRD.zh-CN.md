# 隐信片（Enigma Card）PRD

## 1. 产品概述

### 1.1 产品名称
隐信片（Enigma Card）

### 1.2 产品 Slogan
折叠赛博维度的私语，只给懂的人（与 AI）看。

### 1.3 产品定义
隐信片是一款纯本地运行的轻量化应用，用于将长文本、Prompt、私密内容或叙事指令封装进一张普通图片中。对人类肉眼而言，这是一张极简、克制、甚至近乎空白的“明信片”；对 AI 视觉模型或本地解码器而言，它则是一份可以被展开的隐藏内容载体。

### 1.4 产品使命
在 AI 时代重新发明“隐秘表达”的媒介，让图片不只是视觉对象，也成为可递交给模型阅读的私密信封。

### 1.5 核心原则
- 纯本地运行：编码、解码、存储全部在设备本地完成。
- 离线可用：在无网环境下仍可进行生成、读取和管理。
- 默认私密：不依赖云端，不主动上传用户内容。
- 轻量克制：MVP 聚焦“生成、识读、管理”三件事。
- 高审美门槛：界面和成品都要具备可传播的“物感”和“收藏感”。

## 2. 产品哲学与美学方向

### 2.1 美学隐喻
古董暗格项坠（Locket）。

表面是一件安静、克制、带有情感温度的物件；打开暗格，才显露真正重要的秘密。隐信片延续这一隐喻，让数字图像拥有“表层可见、里层私藏”的双重结构。

### 2.2 视觉关键词
- 极简留白
- 复古胶片
- 古董珠宝纹理
- 低饱和、高质感
- 像私人收藏品而不是工具截图

### 2.3 情绪目标
- 安全感：用户相信内容不会离开自己的设备。
- 仪式感：生成过程像“封蜡”和“装匣”。
- 传播感：成品适合发朋友圈、社媒、论坛、聊天窗口。
- 神秘感：外人只看到一张静默图片，知道的人会把它交给 AI。

## 3. 用户与场景

### 3.1 核心用户

#### A. Prompt 创作者
需要分享长 Prompt、复杂 Skill、工作流说明，但受限于平台字数、折叠、排版和复制丢失。

#### B. 赛博浪漫主义者
需要一种不直白、不廉价、具备情绪张力的表达媒介，用于情书、遗言、日记、哀悼文本、未来留言。

#### C. ARG / 剧本杀 / 互动叙事设计者
需要把线索、任务、密令伪装进图像，制造“被 AI 看见真相”的解谜体验。

### 3.2 用户痛点
- 长文本分享不优雅，复制粘贴容易损坏结构。
- 私密文本直接展示缺少安全感。
- 图片承载信息通常只对人可见，无法为 AI 单独准备隐藏层。
- 现有隐写工具多数偏极客，不具备美学和叙事包装。

### 3.3 典型场景
- 把 3000 字 Prompt 封装成一张卡片发给朋友，对方保存图片后交给 AI 识读。
- 把一封情书封装在一张极简风景图中，公开发图而不公开文字。
- 在游戏中交付一张普通海报，玩家用 AI 或应用内解码器提取秘密任务。

## 4. 产品目标

### 4.1 MVP 目标
- 让用户可以在 3 分钟内生成第一张可用的隐信片。
- 让用户可以离线验证“隐藏内容确实被写入且能被解出”。
- 让用户可以在本地保存和再次打开历史卡片。

### 4.2 非目标（MVP 不做）
- 云同步
- 用户系统与账号体系
- 社区分享广场
- 在线模板商城
- 多人协作
- 复杂图片编辑器
- AI 模型内置调用

## 5. 功能范围

### 5.1 模块 A：铸币局（Encoding）

#### 目标
把用户指定的表面视觉层与隐藏文本层合成为一张高保真 PNG。

#### 输入
- 本地图片上传
- 系统模板选择
- 少量表面引导文字
- 隐藏文本内容
- 编码模式选择

#### 输出
- 一张可下载、可分享、可被本地解码的 PNG 图片

#### 子功能
- 上传封面图
- 选择内置模板
- 输入标题/引导文案
- 输入隐藏文本
- 字数统计与容量提示
- 选择编码方案
- 一键生成与导出
- 生成后本地保存

#### 编码模式
- 微弱色差文本覆盖
- LSB 图像隐写
- 频域隐写（DCT）

#### MVP 推荐默认策略
- 默认使用 LSB 作为稳定模式
- 将“微弱色差注入”作为实验性模式标注
- 将“频域隐写”列为增强模式，放入后续版本验证

原因：LSB 更适合本地可逆验证，也更利于首版保证“可写入、可读出”。微弱色差文本更具概念表达力，但受压缩、显示器和截图链路影响更大，适合作为艺术化增强而非 MVP 主路径。频域隐写在抗压缩方面更强，但工程复杂度、调参与兼容性验证成本更高，更适合作为第二阶段强化能力。

### 5.2 模块 B：显影液（Decoding）

#### 目标
提供产品内“验钞机”能力，验证一张隐信片是否包含隐藏文本，并将其提取展示。

#### 输入
- 导入本地图片

#### 输出
- 提取出的纯文本
- 解码状态与错误提示

#### 子功能
- 图片导入
- 自动识别是否为本产品生成
- 提取文本
- 打字机式逐字展示效果
- 一键复制文本
- 保存为收到的卡片

#### MVP 约束
- 首版仅保证识别本产品生成的标准格式
- 不保证兼容第三方隐写工具

### 5.3 模块 C：手帐本（Gallery）

#### 目标
在本地对草稿、已生成卡片、收到的卡片进行可检索管理。

#### 子功能
- 卡片列表
- 分类筛选：草稿 / 已生成 / 已接收
- 本地缩略图预览
- 查看详情
- 重新导出
- 删除
- 关键词搜索

#### 存储策略
- Web/PWA：IndexedDB
- 桌面端：SQLite 或由 Tauri 插件封装的本地数据库
- 敏感文本在库内加密存储

## 6. 用户流程

### 6.1 生成流程
1. 用户进入铸币局。
2. 选择图片模板或上传图片。
3. 输入表层引导文案。
4. 输入隐藏文本。
5. 系统实时显示容量与兼容性提示。
6. 用户点击生成。
7. 本地完成编码与预览。
8. 用户导出 PNG，并可选择保存至手帐本。

### 6.2 读取流程
1. 用户进入显影液。
2. 导入一张隐信片。
3. 系统识别编码头信息。
4. 若识别成功，提取并展示文本。
5. 用户复制、保存或归档为“收到的卡片”。

### 6.3 管理流程
1. 用户进入手帐本。
2. 浏览历史卡片与草稿。
3. 打开某张卡片查看表层图、隐藏文本摘要、生成时间和编码模式。
4. 用户可再次导出或删除。

## 8. MVP 交互与体验要求

### 8.1 体验基调
- 像使用一件私人器物，而不是一套企业后台
- 操作节奏缓慢但不拖沓
- 强调留白、质地、呼吸感

### 8.2 关键反馈
- 输入过长时，必须明确提示当前图片容量不足
- 生成时展示“封存中”状态而非通用 loading
- 解码失败时说明原因：非本产品图片、图片损坏、编码模式不支持、数据校验失败
- 导出前必须提示“社交平台会压缩图片，建议发送 PNG 原图”

### 8.3 成功反馈
- 生成成功后提供“保存图片”“复制隐藏文本”“存入手帐本”三个动作
- 解码成功后提供“复制文本”“保存到手帐本”两个动作

### 8.4 护城河交互
- 在分享前弹出“请发送原图”提示，明确点名微信、小红书等平台会破坏隐藏信息
- 对高风险分享场景展示压缩风险等级：低风险（原图）、中风险（文件传输）、高风险（社媒转发/截图）
- 对实验性“AI 识读模式”提示：不同模型识别效果不同，优先用产品内解码器验证
- 可选开启“备用密钥徽章”，在卡片角落生成小型二维码或短链标记

说明：二维码/短链只适用于非高度敏感内容，因为这类设计会引入外部检索路径，与“纯本地”哲学存在边界冲突，应作为明确关闭默认值的补充能力。

## 9. 技术方案

## 9.1 推荐路线
优先开发 PWA 版本，再通过 Tauri 封装为桌面应用。

原因：
- PWA 开发与分发成本最低，便于快速验证需求。
- 核心算法依赖 Canvas 与本地存储，天然适合浏览器环境。
- Tauri 封装可复用大部分前端代码，为桌面端补充更强的文件系统能力与本地数据库能力。

### 9.2 技术栈
- 前端框架：React
- UI：Tailwind CSS
- 状态管理：Zustand 或 React Context
- 图像处理：HTML5 Canvas API
- 本地存储：IndexedDB
- 桌面封装：Tauri
- 加密：Web Crypto API

### 9.3 编码格式建议
为保证可解码与可扩展，建议定义自己的封装格式：

- 文件容器：PNG
- 隐藏内容结构：
  - 魔数/标识头
  - 版本号
  - 编码模式
  - 文本长度
  - 压缩标记
  - 校验值
  - 文本正文

### 9.4 算法建议
- 文本先进行 UTF-8 编码
- 可选 gzip/deflate 压缩
- 写入前计算 CRC 校验
- 通过 LSB 将二进制流写入 RGBA 通道的最低位
- 解码时按相同结构逆向恢复

### 9.5 抗压缩与反水印协议

#### 协议目标
提升隐信片在微信、小红书、微博等平台经过 JPG 压缩、截图、局部水印覆盖后的存活率。

#### 战术一：频域隐写
- 放弃仅依赖空间域像素修改，改为将信息注入图像的低频或中频 DCT 系数
- 优势：抗 JPG 压缩能力更强，对平台重采样更有韧性
- 代价：浏览器端实现复杂度高，需要更多图像质量和恢复率验证

#### 战术二：冗余 OCR 注入
- 对面向 AI 视觉模型的可读层，不再只使用极弱色差
- 支持更大的灰阶差值，例如 `#FAFAFA` 对 `#FFFFFF` 或 `#F0F0F0` 对 `#FFFFFF`
- 将同一段文本平铺重复 2 至 3 次，以对抗角落水印或局部裁切
- 配合简单纠错编码，允许部分字符区域受损后仍恢复正文

#### 战术三：交互护栏
- 分享前强制提示“请发送原图”
- 对不同分享方式给出风险说明
- 提供“压缩后预演”能力，模拟社媒压缩后再本地校验一次
- 在非机密场景下，允许附加二维码徽章或短链作为兜底入口

### 9.6 推荐技术分层
- 基础层：LSB 隐写，保证本地稳定编码与解码
- 增强层：冗余 OCR 注入，增强 AI 模型直接识读的概率
- 高级层：DCT 频域隐写，用于抗压缩版本

### 9.7 AI 识读模式说明
- “AI 识读”不等于严格解码，而是利用视觉模型对图像中弱可见结构的理解能力
- 该模式更接近鲁棒视觉提示，而不是标准隐写恢复
- 产品中应与“本地可逆解码”明确区分，避免用户误解

### 9.8 安全边界
- 本地运行不等于绝对安全
- 若图片经过社交平台再次压缩，隐藏信息可能损坏
- 对“AI 一定能读出”的承诺需要谨慎，建议表述为：
  - “适合交给支持视觉理解的模型辅助识读”
  - “产品内置解码器可完整验证内容”
- 二维码徽章、短链备援或云端备份不适合极度机密场景
- 频域抗压缩提升的是鲁棒性，不代表对任意裁切、水印和再编辑都 100% 存活

## 10. 风险与约束

### 10.1 技术风险
- 平台二次压缩可能破坏隐写数据
- 不同图像尺寸对可用容量差异很大
- 微弱色差模式对显示设备和截图链路敏感
- DCT 频域方案需要更复杂的浏览器端实现与充分测试
- 冗余 OCR 模式会牺牲部分视觉纯净度与载荷容量

### 10.2 产品风险
- 用户会把“AI 可读”理解为跨模型、跨平台 100% 兼容
- 用户可能把隐信片当作加密工具，而不是隐写工具
- 用户可能误以为“备用密钥”仍然符合纯本地和绝对私密承诺

### 10.3 应对策略
- 在导出页明确提示“建议使用 PNG 原图传递”
- 提供容量评估与质量警告
- 在设置页和帮助文案中区分“隐写”与“加密”
- 可选提供“本地加密后再隐写”的后续升级路线
- 将“纯本地模式”与“带备援模式”明确区分为两套导出策略
- 用“压缩模拟测试”替代口头承诺，让用户在导出前看到存活概率

## 14. 开发结论

对 MVP 而言，最佳切入点不是“让所有 AI 都能直接看懂隐藏层”，而是：

- 先做一个强可用、强美学、强本地验证的隐写卡片工具
- 用内置解码器确保结果稳定
- 再逐步试验不同视觉注入方案对 AI 模型的可读性

这能最大程度降低首版不确定性，同时保留产品最迷人的概念内核。
