# 隐信片本地网页包分发方案

## 1. 结论

「隐信片」完全可以先做成一版本地网页包，而不必先部署服务器。

推荐把它定义为：

- 纯前端静态应用
- 本地运行
- 可离线使用
- 可后续无缝部署到服务器

这意味着同一套前端代码，既可以：

- 打包后发给别人本地使用
- 也可以在后期直接部署成正式网站

## 2. 推荐分发形态

### 2.1 形态 A：静态网页包

输出内容：
- `index.html`
- `assets/` 目录
- `manifest.webmanifest`
- 图标与静态资源

用户获取方式：
- 下载压缩包
- 解压到本地目录
- 启动一个本地静态服务
- 在浏览器中打开

适合：
- 内测
- 小范围私发
- 极客用户
- 无服务器阶段

### 2.2 形态 B：本地桌面壳

在静态网页包外再套一层桌面壳：
- Tauri
- 或其他超轻量本地容器

用户获取方式：
- 下载安装包
- 双击运行

适合：
- 更强的“像 App 一样”的体验
- 更好地管理本地文件和数据库
- 降低用户对本地服务启动的理解成本

### 2.3 形态 C：正式线上站点

同一套前端打包产物后续可直接部署到：
- Vercel
- Netlify
- Cloudflare Pages
- 自有静态服务器

适合：
- 公开访问
- 产品展示
- 后续接入更多增长能力

## 3. 为什么不建议只靠双击 `index.html`

理论上有些静态页面可以直接通过 `file://` 打开，但对「隐信片」不建议把它作为主要交付方式。

原因：

- `service worker` 通常无法在 `file://` 下正常工作
- PWA 安装能力会受限
- 某些浏览器对本地资源访问策略更严格
- 文件导入、缓存、路径引用的兼容性更差
- 用户遇到问题时更难排查

所以更推荐：

“解压即得 + 一键启动本地服务”

而不是：

“双击 html 直接开”

## 4. 最推荐的本地分发模式

## 4.1 面向内测用户

交付一个压缩包，里面包含：

- `app/`：静态网页产物
- `README`：启动说明
- 本地启动脚本

用户只需要：

1. 解压
2. 双击启动脚本
3. 浏览器自动打开本地地址

## 4.2 面向非技术用户

更适合直接做成 Tauri 桌面版。

原因：
- 他们不想理解“本地服务”是什么
- 他们只想双击就用
- 桌面版更容易承载“本地私密工具”的心智

## 5. 工程架构建议

### 5.1 前端保持纯静态

要求：
- 不依赖后台 API 完成核心功能
- 不依赖服务器存储图片
- 编码、解码、预览、导出全部在浏览器内完成
- 本地数据优先使用 IndexedDB

这会带来两个直接好处：
- 本地网页包可用
- 后续上线网站时无需重写核心逻辑

### 5.2 构建产物保持可托管

要求：
- 构建后只输出静态资源
- 路由尽量采用单页应用可兼容策略
- 资源路径不要写死为特定服务器环境

### 5.3 分发层与产品层分离

建议拆成三层：

- 产品层：React 页面、隐写算法、交互逻辑
- 构建层：Vite 打包输出静态资源
- 分发层：
  - 本地网页包
  - Tauri 桌面壳
  - 正式线上部署

这样未来切换交付方式时，不需要重做产品本体。

## 6. 对「隐信片」的具体适配建议

### 6.1 当前阶段最适合什么

最适合：
- 先做本地网页包
- 同时保留未来部署成网站的能力

原因：
- 产品当前主张“纯本地、绝对隐私、无网可用”
- 本地网页包更符合这个叙事
- 也能避免前期花精力在服务器、账号和后端上

### 6.2 当前阶段不必急着做什么

- 不必先搭后端
- 不必先买服务器
- 不必先做云端数据库
- 不必先做登录体系

## 7. 建议分发目录结构

```text
enigma-card-local/
  app/
    index.html
    assets/
    manifest.webmanifest
    favicon.svg
  scripts/
    start-mac.command
    start-windows.bat
  README.txt
```

说明：
- `app/` 放静态打包产物
- `scripts/` 放本地启动脚本
- `README.txt` 用非常短的语言告诉用户怎么启动

## 8. 用户体验建议

### 8.1 打开方式文案

不要说：
- “请先运行本地 HTTP 静态服务”

建议说：
- “双击启动，浏览器会自动打开隐信片”

### 8.2 产品内提示

在首页或设置页可加入一句说明：

“这是一枚本地运行的隐信片工坊。只要你的浏览器还在，秘密就还在。”

## 9. 后续演进路线

### Phase 1
- 本地网页包
- 小范围私发内测

### Phase 2
- Tauri 桌面壳
- 更完整的离线存储和文件管理

### Phase 3
- 正式部署为线上网站
- 保持纯前端核心不变

## 10. 最终建议

对「隐信片」来说，最佳顺序不是“先上服务器，再考虑本地化”，而是反过来：

1. 先把它做成能本地运行的静态工坊
2. 再把它包成更易用的桌面壳
3. 最后再按需要部署成公开网站

这样最符合产品的隐私叙事，也最节省早期工程成本。
