# 隐信片三端矩阵部署战略

## 1. 核心指导思想

不追求大而全，而是利用不同平台的分发机制、技术边界与用户心智做错位竞争：

- 微信小程序：主攻情绪表达、社交裂变、轻量传播
- 独立网站 / PWA：主攻专业创作、无损导出、纯本地隐私
- Skill 生态：主攻协议扩散、解码标准化、AI 语义延展

一句话总结：

小程序负责让更多人“玩到”，网站负责让专业用户“做深”，Skill 负责让行业“看懂并采用”。

## 2. 三端矩阵总览

### 2.1 微信小程序
- 战略定位：社交裂变场 / 情绪容器
- 目标用户：表白、道歉、生日彩蛋、赛博纪念、轻社交内容创作者
- 核心价值：低门槛生成与分享，让“神秘感”成为传播驱动力
- 核心能力：强提示、强分享、强情绪包装、强审核合规

### 2.2 独立网站 / PWA
- 战略定位：极客避难所 / 专业工坊
- 目标用户：Prompt 工程师、游戏叙事设计师、世界观作者、隐写实验者
- 核心价值：纯本地、无损、可验证、可复现
- 核心能力：LSB 隐写、验钞机式解码、本地管理、无后台依赖

### 2.3 Skill 生态
- 战略定位：万能解码器 / 行业标准
- 目标用户：AI 工具创作者、社区玩家、工作流搭建者、第三方平台作者
- 核心价值：让“隐信片”从单一产品升级成一种可传播的 AI 协议
- 核心能力：统一解码提示词、统一语气、统一结构化输出

## 3. 战线一：微信小程序

### 3.1 定位
小程序不是最强技术阵地，而是最大传播阵地。

它承担的不是“最高保真隐写”，而是“最低门槛参与”和“高情绪触发分享”。用户在小程序里不需要理解隐写原理，只需要感受到：这张图看起来平静，但里面藏着一句只给懂的人看的话。

### 3.2 产品目标
- 让用户在 60 秒内完成一张“可分享”的情绪卡片
- 用情绪场景驱动分享，而不是用技术术语驱动
- 通过明确动效和引导，尽量提升原图发送率

### 3.3 核心产品策略

#### A. 强抗压算法
微信压缩极强，不适合依赖精密 LSB。

小程序版本建议采用：
- 大色差水印隐写
- 冗余排版
- 关键词驱动而非长段逐字精确恢复

设计原则：
- 不使用过于娇贵的微弱色差，如 `#FEFEFE`
- 使用更大灰阶差值，如 `#FAFAFA` 或 `#F0F0F0`
- 将关键文本拆成若干可被 AI 识别的语义锚点，并多次平铺
- 让 AI 通过少量关键词与上下文进行联想补全

#### B. 强制原图引导
生成后必须设计强提示链路：
- 保存到相册
- 动效强调“发送时请勾选原图”
- 分享页二次提醒
- 对高风险路径给出醒目提示，如“朋友圈和截图会吞掉秘密”

#### C. 场景模板优先
小程序优先上线强情绪模板，而不是复杂编辑器：
- 表白卡
- 道歉卡
- 生日彩蛋
- 写给未来
- 纪念与哀悼

### 3.4 技术边界
- 不追求绝对无损
- 不追求标准化可逆解码
- 优先保证“AI 大概率读到情绪核心”
- 页面与生成链路必须足够轻，适应小程序运行限制

### 3.5 风控红线
这是最关键的业务风险。

小程序版本必须接入微信官方安全能力，对：
- 用户输入文本
- 表层文案
- 模板文案
- 分享引导语

进行审核。

建议要求：
- 接入微信小程序官方珊瑚安全 API 或对应内容安全能力
- 对生成行为增加风控策略，如频率限制、可疑词识别、人工复核开关
- 默认拒绝高风险内容，不要用“隐写”作为绕过审核的卖点

结论：
小程序不是“绝对隐私工具”，而是“合规前提下的情绪传播产品”。

## 4. 战线二：独立网站 / PWA

### 4.1 定位
网站是专业版，也是概念最纯粹的一端。

这里承接的是更高阶、更理性、更愿意理解技术边界的用户。他们需要的不是社交裂变，而是可靠的生产环境。

### 4.2 产品目标
- 提供纯前端本地生成和本地解码
- 保证无损 PNG 导出
- 给用户明确、可信的可逆验证能力
- 让界面气质足够克制、专业、可收藏

### 4.3 核心产品策略

#### A. 纯净本地化
- 使用 HTML5 Canvas 完成前端图像合成
- 不依赖后台数据库存储图片
- 默认不上传用户内容
- 可选本地草稿与历史记录，但存储在浏览器本地

#### B. 无损导出
- 采用 LSB 像素级隐写
- 只导出 PNG
- 通过内置验钞机保证“写得进，也读得回”

#### C. 验钞机体验
网站必须有一个“显影液 / 验钞机”功能区：
- 上传隐信片
- 前端解析隐藏内容
- 以电报机或打字机动效逐字显影
- 提供校验结果与复制能力

### 4.4 体验方向
- 视觉上更像古董珠宝鉴定所，而不是在线作图工具
- 文案要节制，少教程腔，多仪式感
- 强调“纯本地”“无网也能工作”“建议原图传递”

### 4.5 技术边界
- 网站端优先承接精密隐写与标准封装格式
- 后续可加压缩模拟、冗余 OCR 实验模式、频域隐写实验区
- 但首版核心永远是：本地稳定闭环

## 5. 战线三：Skill 生态

### 5.1 定位
Skill 不是一个附属物，而是协议扩散器。

它的目标不是替代产品，而是让“隐信片”成为一种 AI 圈内可传播、可复用、可迁移的解码标准。

### 5.2 战略目标
- 让用户把任意隐信片扔给 AI 时，有一套统一提示能引导 AI 识读
- 让不同平台的 AI 工作流都能围绕“隐信片协议”形成兼容层
- 让隐信片的品牌心智从“一个 App”升级成“一种媒介格式”

### 5.3 交付形态
不是先写代码，而是先写一套开源 Prompt / Skill：

- 名称：隐信片解码器
- 发布阵地：GitHub、Coze、Dify、提示词社区、AI 工作流社区
- 使用方式：把图片交给 AI，并调用该 Skill 的解码指令

### 5.4 Skill 应包含的能力
- 指导 AI 识别图片中的隐藏文本、弱可见文字、重复排版块、二维码徽章
- 当内容不完整时，优先保守复述，不擅自脑补机密信息
- 当内容明显是情感文本时，用优美但不过度煽情的方式转述
- 当内容是 Prompt 或结构化指令时，优先保持原始结构、字段和层级
- 当内容带有角色设定时，可在明确提示下继续扮演写信者或任务发布者

### 5.5 Skill 输出原则
- 先说明读到了什么
- 再说明哪些部分确定、哪些部分推测
- 最后根据内容类型决定是转录、总结还是继续互动

### 5.6 战略价值
- 降低用户对特定产品形态的依赖
- 提升隐信片作为协议的可迁移性
- 为后续“暗格集市”或模板生态建立统一语义层

## 6. 三端分工与取舍

### 6.1 小程序不该做什么
- 不做高精度、长篇、无损隐写主战场
- 不宣称绝对隐私
- 不把审核与风控当成可绕过问题

### 6.2 网站不该做什么
- 不做社交流量导向的复杂分享链路
- 不做重后台、重账号体系
- 不为了增长牺牲纯本地和无损导出体验

### 6.3 Skill 不该做什么
- 不承诺所有模型都 100% 读得准
- 不替代产品内置解码器
- 不把推断内容伪装成精确还原

## 7. 推荐推进顺序

### Phase 1
- 先完成网站 / PWA MVP
- 目标：建立标准封装、标准解码、标准体验

### Phase 2
- 推出隐信片解码器 Skill
- 目标：把协议传播出去，开始形成行业语义标准

### Phase 3
- 再做微信小程序
- 目标：把已经验证过的产品心智包装成情绪化、裂变化版本

原因：
- 网站最适合先验证技术闭环
- Skill 最适合低成本放大影响力
- 小程序风控最重、审核最严，应该放在产品语言和协议成熟后再进场

## 8. 给 Codex 的执行映射

### 网站 / PWA
- 继续强化本地 Canvas 编码解码
- 引入 IndexedDB 替代 LocalStorage
- 增加压缩模拟测试与鲁棒性提示
- 保持无后台默认架构

### Skill
- 产出一份开源的《隐信片解码器》提示词规范
- 定义统一输入、输出风格、保守推断原则
- 为 GitHub / Coze / Dify 各准备一份适配版本

### 小程序
- 单独拆分一套大色差冗余排版算法
- 设计强“原图发送”引导链路
- 在立项时同步审核、合规和风控方案

## 9. 最终结论

隐信片不应该只是一款单点工具，而应该形成一个三层结构：

- 网站负责“标准生成”
- Skill 负责“标准解释”
- 小程序负责“标准传播”

当三者联动后，隐信片才真正有机会从一个产品，进化成一种 AI 时代的私密媒介协议。
